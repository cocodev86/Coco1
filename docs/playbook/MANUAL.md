# Metaphor Playbook

**Series:** MPB  
**Version:** 1.0.0  
**Owner:** Operations  
**Status:** Active

## MPB-0001 — Company Operating Manual

Metaphor operates through coordinated consulting, engineering, AI, finance, marketing and knowledge functions.

### Daily cadence
- Review active risks, deadlines and client commitments.
- Prioritize revenue-critical and delivery-critical work.
- Record decisions and blockers in the system of record.
- Use AI to accelerate analysis and production while retaining human accountability.

### Weekly cadence
- Pipeline and revenue review
- Delivery health review
- Product and platform review
- Knowledge capture and documentation review
- Financial and capacity review

### Monthly cadence
- Profitability and cash-flow review
- Client satisfaction and renewal review
- Product portfolio review
- Security, reliability and AI-governance review
- Operating-system improvement review

### Operating rules
- Client commitments must have clear owners and dates.
- Scope changes must be documented and approved.
- Reusable work should be extracted into the Framework when justified.
- Incidents and significant failures require retrospectives.
- Every engagement should improve Metaphor's assets, knowledge or delivery system.

## MPB-0002 — Knowledge Management System

The KMS is the organizational memory for standards, decisions, delivery assets and lessons learned.

### Knowledge classes
- Policies and standards
- Playbooks and procedures
- Product documents
- Architecture decisions
- Technical designs
- Client and project records
- Reusable templates and components
- Research and lessons learned

### Required metadata
Title, owner, status, version, confidentiality, tags, related projects, review date and source authority.

### Lifecycle
Create → Review → Approve → Publish → Discover → Reuse → Revalidate → Archive.

### Search and retrieval
Knowledge should support keyword search, semantic retrieval and relationship-aware discovery. Results must respect tenant and role permissions. AI responses grounded in internal knowledge should identify their sources and freshness.

### Contribution workflow
Contributors draft; domain owners review; authorized owners approve; the Knowledge Curator maintains taxonomy, quality and stale-content reviews.

### Metrics
- Search success rate
- Reuse rate
- Documentation coverage
- Stale-content percentage
- Time to locate authoritative information
- User feedback on usefulness

## MPB-0003 — AI Agent Catalog and Governance

Every AI agent must be registered before production use.

### Agent record
- Name and unique ID
- Purpose and owner
- Model and provider
- Prompt version
- Allowed tools
- Data classifications
- Permissions
- Human approval requirements
- Cost and rate limits
- Evaluation suite
- Monitoring and retirement criteria

### Lifecycle
Propose → Risk assess → Prototype → Evaluate → Approve → Deploy → Monitor → Revalidate → Retire.

### Permission model
Agents receive the minimum tools and data required. Read and write permissions are separated. Financial, legal, security, production and client-facing actions use explicit approval gates proportional to risk.

### Initial catalog
- Architect Agent
- Consultant Agent
- Product Manager Agent
- Designer Agent
- Engineer Agent
- QA Agent
- DevOps Agent
- Knowledge Curator Agent
- Sales Advisor Agent

### Governance
Agent outputs are attributable, observable and reviewable. Material changes to prompts, models, tools or knowledge sources trigger regression evaluation.

## MPB-0004 — Metaphor Delivery System

The MDS is the end-to-end operating model for client and internal delivery.

### Principles
- Human accountability
- AI acceleration
- Reuse before reinvention
- Framework improvement
- Continuous learning
- Transparent client communication

### Stages

1. **Lead** — Create the opportunity record and capture the initial need.
2. **Discovery** — Understand business context, users, systems, constraints and desired outcomes.
3. **Assessment** — Evaluate feasibility, maturity, risk, architecture and automation potential.
4. **Proposal** — Define outcomes, scope, assumptions, timeline, pricing and terms.
5. **Planning** — Produce the roadmap, milestones, staffing plan, risk register and delivery controls.
6. **Design** — Complete product, UX, architecture, data, security and AI designs.
7. **Build** — Implement in controlled increments using shared capabilities.
8. **Verification** — Test functionality, accessibility, security, performance and AI behavior.
9. **Deployment** — Release with monitoring, rollback and stakeholder communication.
10. **Support** — Stabilize, resolve incidents and honor service commitments.
11. **Optimization** — Measure outcomes, identify improvements and expand proven value.

### Human responsibilities
Humans own commercial commitments, architecture approvals, client communication, high-risk decisions, acceptance and final accountability.

### AI responsibilities
AI may research, summarize, generate drafts, recommend designs, produce code, create tests, analyze results and automate routine operations within governed limits.

### Stage gates
Each stage requires defined deliverables, owner approval and unresolved-risk visibility. Exceptions are documented with an owner, rationale and expiration.

### Delivery metrics
- Lead-to-proposal time
- Proposal win rate
- Milestone predictability
- Cycle time
- Defect escape rate
- Gross margin
- Framework reuse
- Client satisfaction
- Time to value

### Knowledge capture
At engagement close, record outcomes, reusable assets, architecture decisions, failure modes, client feedback and recommended changes to the MOS, Playbook or Framework.
